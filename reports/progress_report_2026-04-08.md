## 2. Problem Diagnosis
### Symptom
- GitHub Actions workflows were failing at the `Commit + Push Data` step with "failed to push some refs" errors.
- Scheduled runs at `:00`, `:15`, and `:30` past the hour were clashing, as each attempted to modify the same large file.
- The standard 5-retry loop was insufficient to overcome persistent merge conflicts on the shared JSON state.

### Root Cause: "Conflict Storm"
Every sector workflow (Environment, Emergency, Industry, etc.) appends new data to the *top* of `data/latest_intel.json`. When multiple workflows run concurrently, Git cannot automatically merge these simultaneous insertions into the same JSON array boundary. This created a persistent conflict that manual `git pull --rebase` could not resolve automatically in a headless CI environment.

## 3. Implemented Solutions

### A. Smart JSON Merge (The "Self-Healer")
We introduced `scripts/resolve_conflicts.py`, a robust Python-based Git resolver.
- **Function**: During a `git pull --rebase`, if a conflict is detected in a `.json` file, the script intercepts the process.
- **Logic**: It parces the conflicting versions, merges the data arrays, deduplicates by the unique `hash` field, and sorts the final set by timestamp.
- **Result**: Workflows no longer stall on data collisions; they merge the remote data into their local commit and proceed.

### B. Final Hardening Pass
- **Retry Logic**: Increased from 5 to **15 retries** with expanded randomized jitter (**5–25s**) to handle extreme concurrency bursts.
- **Observability**: Upgraded scraper logging to **GitHub Annotations** (`::warning::`), ensuring that individual feed failures are visible on the Action Summary page without digging into raw logs.
- **Schedule Staggering**: Completely de-collided the `:00` crons, distributing sector updates across the hour to minimize lock contention.
